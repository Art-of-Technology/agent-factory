# Architecture Defaults

These are the **default architectural decisions** for all projects bootstrapped with Agent Factory. Agents MUST follow these unless explicitly overridden by the project's `PROJECT.md` or setup configuration.

---

## Package Manager — Bun (MANDATORY)

**Always use `bun`** as the package manager and runtime. Never use npm, pnpm, yarn, or npx.

| Instead of | Use |
|---|---|
| `npm install` / `pnpm install` | `bun install` |
| `npm run dev` | `bun run dev` |
| `npx` / `pnpx` | `bunx` |
| `node script.js` | `bun script.js` |
| `tsx watch src/index.ts` | `bun --watch src/index.ts` |
| `ts-node` / `tsx` | `bun` (native TS support) |

### Monorepo Workspaces
Use bun workspaces in root `package.json` (NOT pnpm-workspace.yaml):
```json
{
  "workspaces": ["apps/*", "packages/*", "tooling/*"]
}
```

### Lock File
- Use `bun.lockb` — never commit `package-lock.json`, `pnpm-lock.yaml`, or `yarn.lock`
- Add those to `.gitignore`

### Scripts in package.json
```json
{
  "scripts": {
    "dev": "bun run --filter './apps/*' dev",
    "build": "bun run --filter './apps/*' build",
    "db:generate": "bunx drizzle-kit generate",
    "db:migrate": "bunx drizzle-kit migrate",
    "db:seed": "bun packages/db/src/seed.ts"
  }
}
```

### Why Bun
- Fastest JS runtime + package manager
- Native TypeScript support (no tsx/ts-node needed)
- Compatible with Node.js APIs
- Faster installs, faster builds, faster everything

## App Separation (Monorepo — Turborepo)

Every project follows a **3-app monorepo** pattern:

```
{REPO}/
├── apps/
│   ├── web/              → Public-facing web app (Next.js 15)
│   ├── admin/            → Admin dashboard/platform (Next.js 15)
│   └── api/              → API server (Hono on Node/Bun)
├── packages/
│   ├── auth/             → Better Auth config + adapters
│   ├── db/               → Drizzle ORM + schema + migrations
│   ├── ui/               → Shared UI components (Tailwind + shadcn/ui)
│   ├── shared/           → Shared types, utils, constants
│   └── email/            → Email templates (React Email) — optional
├── tooling/
│   ├── eslint/           → Shared ESLint config
│   ├── typescript/       → Shared tsconfig
│   └── tailwind/         → Shared Tailwind config
├── turbo.json
├── package.json
└── PROJECT.md            → Project-specific context (generated by setup)
```

### App Responsibilities

| App | Purpose | URL Pattern | Dev Port |
|-----|---------|-------------|----------|
| **web** | Customer-facing product UI, public pages, KYC flows, onboarding | `app.{domain}` | `4100` |
| **admin** | Internal admin dashboard, user management, org management, analytics, config | `admin.{domain}` | `4200` |
| **api** | REST API, webhooks, background jobs, auth endpoints, file uploads | `api.{domain}` | `4300` |

### Port Configuration (IMPORTANT — avoid conflicts)
**DO NOT use default ports** (3000, 3001, 3002, 5432) — they are used by production services.

| Service | Default Port | Our Port |
|---------|-------------|----------|
| web app | 3000 | **4710** |
| admin app | 3001 | **4720** |
| api server | 3003 | **4730** |
| PostgreSQL | 5432 | **5491** |

Configure in each app's `package.json`:
```json
{
  "scripts": {
    "dev": "next dev -p 4710"
  }
}
```

For Hono API (`apps/api/package.json`):
```json
{
  "scripts": {
    "dev": "bun --watch src/index.ts"
  }
}
```
With port set in code: `serve({ fetch: app.fetch, port: 4730 })`

For PostgreSQL in `docker-compose.yml` or `.env`:
```env
DATABASE_URL=postgresql://maestro:maestro@localhost:5491/maestro_kyc
```

### Rules
- Apps import from `packages/` — never from other apps
- Each app has its own `package.json` and can be deployed independently
- Shared business logic lives in `packages/shared`
- Database access ONLY through `packages/db` (never raw SQL in apps)
- **NEVER use default ports** — always use the configured dev ports above

## Authentication (Better Auth)

**Library**: [Better Auth](https://www.better-auth.com/) — the default auth solution.

### Setup
- Auth config lives in `packages/auth/`
- Server-side auth instance created with `betterAuth()`
- Client-side auth hooks via `createAuthClient()`
- Session management via secure HTTP-only cookies

### Providers
- **Google OAuth** (primary) — configured via `GOOGLE_CLIENT_ID` + `GOOGLE_CLIENT_SECRET`
- **Email + Password** (secondary) — optional, enable per project
- **Magic Link** (optional) — if email provider is configured

### Auth Flow
1. User clicks "Sign in with Google" → OAuth flow
2. Better Auth creates/links account automatically
3. Session cookie set → user redirected to app
4. Admin app has separate auth check (requires admin role)

### Environment Variables
```env
BETTER_AUTH_SECRET=<random-32-char-string>
BETTER_AUTH_URL=https://api.{domain}
GOOGLE_CLIENT_ID=<from-google-console>
GOOGLE_CLIENT_SECRET=<from-google-console>
```

## Multi-Tenancy

### Core Concept
- **One account → multiple organizations**
- A user signs up once and can create or join multiple orgs
- Each org is an isolated tenant with its own data, roles, and settings
- User switches between orgs via org switcher in the UI

### Database Schema Pattern
```
User (1) ──→ (N) OrgMembership (N) ←── (1) Organization
                    │
                    ├── role: Role
                    ├── joinedAt: DateTime
                    └── invitedBy: User (optional)
```

### Rules
- **Every data query MUST be scoped to orgId** — no cross-tenant data leaks
- Middleware validates `orgId` on every API request
- User's "active org" stored in session/cookie
- Org creation is free (no approval needed)
- Org deletion is soft-delete with 30-day grace period

### Better Auth Multi-Tenant Plugin
Use Better Auth's built-in `organization` plugin:
```typescript
import { betterAuth } from 'better-auth'
import { organization } from 'better-auth/plugins'

export const auth = betterAuth({
  plugins: [
    organization({
      allowUserToCreateOrganization: true,
    })
  ]
})
```

## Scopes & Roles (RBAC)

### Architecture
- **Scopes** are atomic permissions (e.g., `users:read`, `users:write`, `billing:manage`, `kyc:review`)
- **Roles** are collections of scopes — admin creates roles by picking scopes
- Roles are **per-organization** (each org can have different roles)
- Some system roles are built-in: `owner`, `admin`, `member`

### Scope Format
```
resource:action
```

### Default Scopes (seeded via DB seed)

All scopes are seeded on first run. Admin can create new roles by picking from these scopes.

```
# ── Users ──
users:read              View user list and profiles
users:write             Create and edit users
users:delete            Remove users from organization

# ── Organizations ──
orgs:read               View organization details
orgs:write              Edit organization settings
orgs:delete             Delete organization
orgs:billing            Manage billing and subscription

# ── Team / Members ──
members:read            View team member list
members:invite          Create and share invite links
members:remove          Remove members from organization
members:role            Change member roles

# ── Roles & Permissions ──
roles:read              View roles and their scopes
roles:write             Create and edit custom roles
roles:delete            Delete custom roles

# ── KYC ──
kyc:submit              Submit KYC verification
kyc:read                View KYC submissions
kyc:review              Review and approve/reject KYC
kyc:admin               Full KYC management (config, templates)

# ── Reports & Analytics ──
reports:read            View reports and dashboards
reports:export          Export reports (CSV, PDF)
analytics:read          View analytics data

# ── Settings ──
settings:read           View organization settings
settings:write          Change organization settings
settings:integrations   Manage third-party integrations
settings:api            Manage API keys and webhooks

# ── Audit ──
audit:read              View audit logs
```

### Default Roles (seeded via DB seed)

These system roles are created on seed. They CANNOT be deleted (isSystem=true) but admin can create additional custom roles.

| Role | Scopes | System? |
|------|--------|---------|
| **Owner** | ALL scopes (wildcard `*`) | ✅ |
| **Admin** | Everything except `orgs:delete`, `orgs:billing` | ✅ |
| **Manager** | `users:read`, `members:*`, `roles:read`, `kyc:read`, `kyc:review`, `reports:*`, `analytics:read`, `settings:read`, `audit:read` | ✅ |
| **Member** | `users:read`, `members:read`, `kyc:submit`, `kyc:read`, `reports:read`, `settings:read` | ✅ |
| **Viewer** | `users:read`, `members:read`, `reports:read`, `analytics:read`, `settings:read` | ✅ |
| **KYC Reviewer** | `users:read`, `kyc:read`, `kyc:review`, `reports:read` | ✅ |

### DB Seed Script (`packages/db/seed.ts`)

The seed script MUST:
1. Insert all default scopes into `scopes` table (idempotent — skip if exists)
2. Insert all default roles with their scope arrays (idempotent)
3. Mark default roles as `isSystem: true`
4. Be runnable multiple times safely (upsert pattern)

```typescript
// packages/db/seed.ts
import { db } from './index'
import { scopes, roles } from './schema'

const DEFAULT_SCOPES = [
  { key: 'users:read', label: 'View users', category: 'Users', description: 'View user list and profiles' },
  { key: 'users:write', label: 'Create/edit users', category: 'Users', description: 'Create and edit users' },
  { key: 'users:delete', label: 'Delete users', category: 'Users', description: 'Remove users from organization' },
  // ... all scopes listed above
]

const DEFAULT_ROLES = [
  { name: 'Owner', scopes: ['*'], isSystem: true, description: 'Full access to everything' },
  { name: 'Admin', scopes: ['users:*', 'members:*', 'roles:*', 'kyc:*', 'reports:*', 'analytics:*', 'settings:*', 'audit:*'], isSystem: true, description: 'Organization administrator' },
  { name: 'Manager', scopes: ['users:read', 'members:*', 'roles:read', 'kyc:read', 'kyc:review', 'reports:*', 'analytics:read', 'settings:read', 'audit:read'], isSystem: true, description: 'Team and KYC manager' },
  { name: 'Member', scopes: ['users:read', 'members:read', 'kyc:submit', 'kyc:read', 'reports:read', 'settings:read'], isSystem: true, description: 'Standard team member' },
  { name: 'Viewer', scopes: ['users:read', 'members:read', 'reports:read', 'analytics:read', 'settings:read'], isSystem: true, description: 'Read-only access' },
  { name: 'KYC Reviewer', scopes: ['users:read', 'kyc:read', 'kyc:review', 'reports:read'], isSystem: true, description: 'Can review and approve KYC submissions' },
]

async function seed() {
  // Upsert scopes
  for (const scope of DEFAULT_SCOPES) {
    await db.insert(scopes).values(scope).onConflictDoNothing({ target: scopes.key })
  }
  // Upsert roles (org-level roles are created per-org, system roles are global templates)
  for (const role of DEFAULT_ROLES) {
    await db.insert(roles).values(role).onConflictDoNothing({ target: [roles.name, roles.isSystem] })
  }
}
```

### Admin Role Management UI
- Admin sees a list of all roles (system + custom)
- System roles show a 🔒 badge — can view but not delete
- "Create Role" button opens a form:
  - Role name + description
  - Scope picker: grouped by category (Users, KYC, Reports, etc.)
  - Each scope is a toggle/checkbox with label + description
  - Preview section: "This role can: View users, Submit KYC, View reports"
- Edit custom roles: same form, pre-filled
- Delete custom roles: confirmation dialog (only non-system roles)
- Wildcard support: `*` means all scopes, `users:*` means all user scopes

### Database Schema
```
Scope (reference table — seeded)
  ├── id
  ├── key            (unique, e.g., "kyc:review")
  ├── label          (human-readable, e.g., "Review KYC")
  ├── category       (grouping, e.g., "KYC", "Users", "Reports")
  ├── description
  └── createdAt

Role
  ├── id
  ├── orgId          → Organization (null for system/template roles)
  ├── name           (e.g., "KYC Reviewer")
  ├── description
  ├── scopes         (string[] — list of scope keys, supports wildcards: "*", "users:*")
  ├── isSystem       (boolean — system roles can't be deleted or renamed)
  ├── createdAt
  └── updatedAt

OrgMembership
  ├── userId
  ├── orgId
  ├── roleId         → Role
  └── customScopes   (string[] — additional scopes beyond role, optional)
```

### Scope Resolution Logic
When checking permissions, resolve scopes in this order:
1. Get role scopes from `role.scopes[]`
2. Add `membership.customScopes[]` (if any)
3. Resolve wildcards: `*` = all, `users:*` = all user scopes
4. Check if requested scope is in the resolved set

### Admin UI for Role Management
- Admin can create custom roles by toggling scopes on/off
- Scopes displayed in grouped categories (Users, Billing, KYC, etc.)
- Each scope has a human-readable label + description
- Preview: "This role can: View users, Submit KYC, View reports"

### Middleware Pattern
```typescript
// API middleware checks scope
function requireScope(scope: string) {
  return async (ctx) => {
    const member = await getOrgMembership(ctx.user.id, ctx.orgId)
    const role = await getRole(member.roleId)
    const allScopes = [...role.scopes, ...(member.customScopes || [])]
    if (!allScopes.includes(scope)) throw new ForbiddenError()
  }
}

// Usage
app.get('/api/users', requireScope('users:read'), listUsers)
app.post('/api/users', requireScope('users:write'), createUser)
```

## Team Invitation System (Link-Based, No Email)

### How It Works
1. **Admin/Owner generates an invite link** from the admin panel
2. Link format: `https://app.{domain}/invite/{inviteCode}`
3. **Invite code** is a unique, URL-safe token (nanoid, 21 chars)
4. Anyone with the link can join — **no email verification required**
5. New users sign up via Google OAuth, then auto-join the org
6. Existing users click link → auto-join the org

### Invite Properties
```
Invite
  ├── id
  ├── orgId           → Organization
  ├── code            (unique, URL-safe token)
  ├── roleId          → Role (what role they'll get)
  ├── createdBy       → User (who created the invite)
  ├── maxUses         (null = unlimited)
  ├── usedCount       (how many times used)
  ├── expiresAt       (null = never expires)
  ├── isActive        (can be deactivated)
  ├── createdAt
  └── updatedAt
```

### Rules
- **No email sending** — invites are purely link-based
- Admin can set: role, max uses, expiration
- Admin can deactivate/delete invite links anytime
- Each invite tracks who used it and when
- Org owner can create multiple invite links with different roles
- Invite page shows org name + "Join as [role]" before accepting

### UI Flow
1. Admin → Team Settings → "Create Invite Link"
2. Select role for invitees
3. Optional: set max uses, expiration
4. Generate → copy link → share via any channel (Slack, WhatsApp, etc.)
5. Recipient opens link → signs in with Google → joins org with assigned role

## Frontend Design Standards

### Philosophy (from Anthropic's Frontend Design Skill)
Create distinctive, production-grade frontend interfaces that avoid generic "AI slop" aesthetics. Every project should have a BOLD, intentional design direction.

### Before Coding Any UI
1. **Define the aesthetic direction** — don't default to generic. Choose: brutally minimal, luxury/refined, editorial, playful, industrial, retro-futuristic, etc.
2. **Typography matters** — never use Inter, Roboto, Arial, or system fonts as primary. Choose distinctive fonts from Google Fonts that match the project personality.
3. **Color with conviction** — dominant colors with sharp accents. No timid, evenly-distributed palettes.
4. **Motion is delight** — page load animations, staggered reveals, hover states that surprise. Use CSS transitions/animations, Framer Motion for complex.
5. **Spatial composition** — asymmetry, overlap, generous negative space OR controlled density. Break the grid sometimes.

### Never Do
- Purple gradients on white backgrounds (AI slop signal #1)
- Inter/Roboto/Arial as primary font
- Cookie-cutter card grids with even spacing
- Generic dashboards that look like every SaaS template
- Predictable layouts without any visual tension

### Always Do
- Pick fonts that have CHARACTER
- Use CSS variables for theming consistency
- Add background textures, gradients, or patterns that create atmosphere
- Include micro-interactions on hover/focus/active states
- Dark mode support from day one
- Mobile-first responsive design

## Database (Drizzle ORM + PostgreSQL)

### Why Drizzle (not Prisma)
- Lighter weight, faster cold starts
- SQL-like syntax (less abstraction)
- Better edge/serverless compatibility
- Schema-as-code in TypeScript

### Schema Location
- `packages/db/schema/` — one file per domain (users.ts, orgs.ts, kyc.ts, etc.)
- `packages/db/migrations/` — auto-generated by `drizzle-kit`
- `packages/db/index.ts` — exports db client + all schemas

### Conventions
- Use `uuid` for primary keys (not auto-increment)
- Always include: `id`, `createdAt`, `updatedAt`
- Soft deletes: `deletedAt` column (nullable timestamp)
- Foreign keys with `onDelete: 'cascade'` or `'set null'` — document which and why
- Indexes on all foreign keys + frequently queried columns

## API Design (Hono)

### Why Hono
- Lightweight, fast, edge-compatible
- Built-in middleware ecosystem
- TypeScript-first with type-safe routes
- Works on Node, Bun, Cloudflare Workers, Deno

### Structure
```
apps/api/
├── src/
│   ├── index.ts              → App entry, middleware stack
│   ├── routes/
│   │   ├── auth.ts           → Auth routes (Better Auth handler)
│   │   ├── users.ts          → User CRUD
│   │   ├── orgs.ts           → Organization management
│   │   ├── invites.ts        → Invite link management
│   │   ├── roles.ts          → Role/scope management
│   │   └── kyc.ts            → KYC-specific routes
│   ├── middleware/
│   │   ├── auth.ts           → Session validation
│   │   ├── org.ts            → Org context + tenant scoping
│   │   ├── rbac.ts           → Scope/permission checks
│   │   └── rate-limit.ts     → Rate limiting
│   └── lib/
│       ├── errors.ts         → Custom error classes
│       └── validators.ts     → Zod schemas
```

### Conventions
- All routes return JSON with consistent shape: `{ data }` or `{ error: { code, message } }`
- Input validation with Zod on every endpoint
- Auth middleware on all routes except public ones
- Org middleware injects `orgId` into context
- RBAC middleware checks scopes per-route
